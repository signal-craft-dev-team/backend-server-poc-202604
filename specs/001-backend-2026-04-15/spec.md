# Feature Specification: Signal Craft Backend POC

**Feature Branch**: `001-backend-2026-04-15`
**Created**: 2026-04-15

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 현장 장비 등록 (Priority: P1)

현장에 새로 설치된 엣지 서버 또는 엣지 센서가 시스템에 자신을 등록하면,
백엔드는 해당 장비를 식별하고 등록 결과를 장비에 전달한다.
이후 모든 통신 및 제어는 등록된 장비만 허용된다.

**Why this priority**: 장비 등록이 선행되지 않으면 오디오 수집과 파라미터 제어가 불가능하다.
모든 시나리오의 전제 조건이므로 가장 먼저 구현되어야 한다.

**Independent Test**: 새 엣지 서버와 새 엣지 센서 각각의 등록 요청을 처리하고,
DB에 장비 정보가 기록되며 등록 결과 메시지가 장비에 전달되는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 미등록 엣지 서버가 등록을 요청할 때,
   **When** 백엔드가 요청을 수신하면,
   **Then** 백엔드는 장비를 DB에 등록하고 성공 결과를 엣지 서버에 전달한다.

2. **Given** 등록된 엣지 서버가 연결된 미등록 엣지 센서의 등록을 중계할 때,
   **When** 백엔드가 중계 요청을 수신하면,
   **Then** 백엔드는 센서를 해당 엣지 서버와 연결하여 DB에 등록하고,
   결과를 엣지 서버로 전달한다.

3. **Given** 이미 등록된 장비가 재등록을 요청할 때,
   **When** 백엔드가 요청을 수신하면,
   **Then** 기존 등록 정보를 유지하고 중복 등록 없이 결과를 전달한다.

---

### User Story 2 - 현장 장비 파라미터 원격 제어 (Priority: P2)

운영자가 엣지 서버 또는 엣지 센서의 동작 파라미터를 원격으로 변경하면,
백엔드가 해당 장비에 변경값을 전달하고 적용 결과를 수신하여 확인한다.

**Why this priority**: 헌장 원칙 IV에 따라 CTRL은 AUDIO보다 먼저 구현한다.
장비 등록 직후 동작 상태 조정이 필요하며, 파라미터 제어 경로 검증이 오디오 수집보다 선행되어야 한다.

**Independent Test**: 엣지 서버 대상 파라미터 변경 명령을 전송하고 성공 결과가 수신되는지,
엣지 센서 대상 파라미터 변경 명령이 엣지 서버를 통해 센서까지 전달되고 결과가 백엔드까지 올라오는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 운영자가 엣지 서버의 파라미터 변경을 요청할 때,
   **When** 백엔드가 변경 명령을 엣지 서버에 전달하면,
   **Then** 엣지 서버는 파라미터를 적용하고 결과를 백엔드로 반환한다.

2. **Given** 운영자가 엣지 센서의 파라미터 변경을 요청할 때,
   **When** 백엔드가 엣지 서버를 통해 센서에 변경 명령을 전달하면,
   **Then** 센서는 파라미터를 적용하고 결과가 엣지 서버를 거쳐 백엔드에 전달된다.

3. **Given** 장비가 파라미터 변경에 실패할 때,
   **When** 실패 결과가 백엔드에 전달되면,
   **Then** 백엔드는 실패 상태를 기록하고 운영자가 상황을 파악할 수 있도록 한다.

### User Story 3 - 오디오 데이터 클라우드 저장 지원 (Priority: P3)

현장의 엣지 서버가 수집된 오디오 데이터를 클라우드 스토리지에 업로드할 수 있도록
백엔드가 안전한 업로드 경로를 발급하고, 업로드 완료 후 결과를 기록한다.

**Why this priority**: 오디오 수집이 이 시스템의 핵심 가치이나,
장비 등록과 파라미터 제어 경로가 검증된 이후 단계에서 구현한다.

**Independent Test**: 등록된 엣지 서버의 업로드 요청에 대해 유효한 업로드 경로가 발급되고,
업로드 완료 통보 수신 시 DB에 수집 이력이 기록되는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 등록된 엣지 서버가 오디오 업로드를 요청할 때,
   **When** 백엔드가 요청을 수신하면,
   **Then** 15분 동안 유효한 업로드 경로(Presigned URL)를 발급하여 엣지 서버에 전달한다.

2. **Given** 엣지 서버가 업로드 완료를 백엔드에 알릴 때,
   **When** 백엔드가 완료 통보를 수신하면,
   **Then** 해당 오디오의 수집 이력(장비, 시각, 저장 위치 등)을 DB에 기록한다.

3. **Given** 유효 시간이 만료된 업로드 경로로 업로드가 시도될 때,
   **When** 엣지 서버가 업로드 완료를 보고할 때,
   **Then** 백엔드는 실패 이력을 기록하고 적절한 결과를 반환한다.

4. **Given** 엣지 서버가 업로드 완료 통보를 보내지 않을 때,
   **When** 발급된 업로드 경로(Presigned URL)의 유효 시간(15분)이 만료되면,
   **Then** 백엔드는 미완료 이력을 DB에 기록하고 별도 조치를 취하지 않는다.

---

### Edge Cases

- 등록되지 않은 장비가 오디오 업로드 또는 파라미터 제어를 요청하면, 백엔드는 요청을 무시하고 경고 로그를 기록한다 (응답 없음).
- 파라미터 변경 명령 전달 후 장비로부터 응답이 없을 경우, 백엔드는 30초 타임아웃 후 실패 이력을 기록한다 (재시도 없음).
- 오디오 업로드 경로 발급 후 장비가 업로드 완료 통보를 보내지 않으면, 유효 시간 만료 후 미완료 이력을 DB에 기록하고 별도 조치는 취하지 않는다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 미등록 엣지 서버의 등록 요청을 수신하고 DB에 기록해야 한다.
- **FR-002**: 시스템은 엣지 서버를 통해 중계된 엣지 센서 등록 요청을 수신하고, 해당 엣지 서버와 연결하여 DB에 기록해야 한다.
- **FR-003**: 시스템은 등록 처리 결과(성공/실패)를 요청 장비에 전달해야 한다.
- **FR-004**: 시스템은 등록된 엣지 서버의 오디오 업로드 요청을 수신하고, 15분 동안 유효한 업로드 경로(Presigned URL)를 발급하여 전달해야 한다.
- **FR-005**: 시스템은 엣지 서버로부터 오디오 업로드 완료 통보를 수신하고, 수집 이력을 DB에 기록해야 한다.
- **FR-006**: 시스템은 엣지 서버 파라미터 변경 명령을 엣지 서버에 전달하고 결과를 수신해야 한다.
- **FR-007**: 시스템은 엣지 센서 파라미터 변경 명령을 엣지 서버를 통해 전달하고 최종 결과를 수신해야 한다.
- **FR-008**: 시스템은 모든 메시지 송수신을 비동기적으로 처리해야 하며, 하나의 요청 처리가 다른 요청에 영향을 주어서는 안 된다.
- **FR-009**: 시스템은 미등록 장비로부터의 파라미터 제어 또는 오디오 업로드 요청을 무시하고 경고 로그를 기록해야 한다. 해당 장비에는 응답을 전달하지 않는다.

### Key Entities

- **엣지 서버**: 현장에 설치된 게이트웨이 장비. 식별자, 설치 위치, 등록 상태, 등록 시각을 가진다.
- **엣지 센서**: 현장에서 오디오를 수집하는 장비. 식별자, 연결된 엣지 서버, 등록 상태를 가진다.
- **오디오 수집 이력**: 수집된 오디오의 메타데이터. 수집 장비, 수집 시각, 저장 위치, 업로드 결과를 가진다.
- **파라미터 제어 이력**: 파라미터 변경 명령의 대상 장비, 변경값, 전달 결과, 처리 시각을 가진다.

## Success Criteria *(mandatory)*

- **SC-001**: 새 엣지 서버 등록 요청부터 결과 전달까지 전체 흐름이 끊김 없이 완결된다.
- **SC-002**: 새 엣지 센서 등록 요청이 백엔드에서 처리되어 결과가 엣지 서버와 센서까지 전달된다.
- **SC-003**: 오디오 업로드 경로 발급 요청이 수신된 시점으로부터 경로가 전달되기까지 5초 이내에 완료된다.
- **SC-004**: 업로드 완료 통보 수신 시 수집 이력이 누락 없이 DB에 기록된다.
- **SC-005**: 파라미터 변경 명령이 엣지 서버 또는 센서에 전달되고 결과가 백엔드까지 전파된다. 장비 응답이 없을 경우 30초 이내에 타임아웃 처리되어 실패 이력이 기록된다.
- **SC-006**: 여러 장비에서 동시에 요청이 들어오더라도 각 요청이 독립적으로 처리되어 상호 간섭이 없다.

- Q: 파라미터 변경 명령의 타임아웃 시간은? → A: 30초
- Q: 오디오 업로드 경로(Presigned URL)의 유효 시간은? → A: 15분

## Assumptions

- 클라우드 브로커(MQTT)는 이미 운영 중이며, 백엔드는 이를 활용하여 엣지 서버와 통신한다.
- 엣지 브로커는 현장의 엣지 서버에 Docker로 함께 기동되며, 백엔드는 엣지 브로커에 직접 접근하지 않는다.
- 오디오 업로드는 클라우드 스토리지의 Presigned URL 방식을 사용하며, 실제 오디오 바이너리는 백엔드를 경유하지 않는다.
- GCP VM에서 서비스 계정이 직접 연결되어 있어 별도 자격증명 파일 없이 클라우드 스토리지 접근이 가능하다.
- 운영자의 파라미터 제어 요청이 어떤 경로(관리 UI, 직접 API 호출 등)로 들어오는지는 이번 POC 범위 밖이다.
  현재는 백엔드가 제어 명령을 장비에 전달하고 결과를 수신하는 부분만 구현한다.
- 오디오 업로드 후 AI 분석 등 다운스트림 처리는 이번 POC 범위 밖이다.
